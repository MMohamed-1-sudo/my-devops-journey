WHAT IS VERSION CONTROL?
    - Version control tracks changes to code overtime
    - Allows you to undo your mistakes, see who changed what and collaborate without stepping on each others toes. 
    - In Devops it isnt just about application code, this also works for terraform files, modules kubernetes and manifests, Docker files, CICD configs all these things can reside in git and can be 
    modified and controlled by it.
    - Git allows you to go back in time, see the history review it and revert if you wish to do so.
    - essenstially git allows for a safe space to experiment, review and recover any code sort of like a time machine for your infrastructure and your code.
    - In modern Devops Git is where everything starts such as pipelines deployemnts, infra all of usaually kicking of from from acute repository.

CENTRALISED (SVN) VS DISTRIBUTED VCS (Git)

CENTRALISED 
    - single source of truth 
    - requires network to commit 
    - Conflicts are common
    - server outages meant that you could commit, you couldnt update and couldnt work basically stuck.
    - you had to lock files during edits 
    - collaberation was linear and tightly controlled meaning only one person at a time could wor on a single file making thing slow.

In 2005 linus tervold decided to write his own version control system, few years later Git was born, instead of relying on one server every developer hjas full copy of the repo 
include the entire history.

DISTRIBUTED (Git)
    - allowing full history on every machine 
    - git allows you to work offline 
    - git allows you to branch as much as you want and merge changes later 
    - not affected by outages 
    - everyone can commit locally ( every commit is a snapshot if your mess something up you justbrewind like a time machine ).

Think of Git like google docks one master file that everyoone can edit if it down noone can save it, gt is like having your own copy of that doc witha full history,
 you work independantly then moerge your updates when your ready.
